Ngày 03: Agent Execution Harness (PAL & PAM)

Phân tích cơ chế thực thi Agent ngầm (under the hood): chu kỳ điều phối (orchestration loop), giao tiếp công cụ (tool interface), quản lý ngữ cảnh/bộ nhớ, hệ thống kiểm duyệt (guardrails) và điểm can thiệp của con người (HITL) dựa trên các bằng chứng quan sát thực nghiệm.

1 PHẦN 1: AGENT EXECUTION HARNESS DIAGRAM (Sơ đồ Kiến trúc Vận hành)

Sơ đồ mô tả "bộ khung" thực thi (Harness) bao bọc quanh AI Agent. Nó điều hướng dòng chảy dữ liệu, tiêm tri thức (Context Injection), kích hoạt công cụ (Tools), và kiểm duyệt đầu ra (Evaluator Guardrail) trước khi phản hồi về Client.

Giai đoạn 1: Input & Gatekeeping (PAM Client Gate)
User Prompt / Brief Nhận đầu vào từ Marketer và các thẻ tệp đính kèm.
Input Guardrail Bộ lọc Regex/Classifier kiểm duyệt từ cấm ở Gateway (trả về lỗi < 200ms).
Context Resolver Truy xuất sliding window chat và tìm kiếm RAG từ thẻ `@logo`.
POST Request (Payload JSON)
Giai đoạn 2: Handoff & Dispatching (PAL Webhook Trigger)
API Endpoint Resolver Tiếp nhận Payload từ PAM, phân tích token và phân quyền.
DAG Workflow Router Ánh xạ Brief và Playbook ID sang đồ thị xử lý AI (DAG) tương ứng.
DAG Execution Pipeline
Giai đoạn 3: AI Orchestrator & Tool Call Loop (PAL Core)
LLM Agent Node (Generator) Nạp System Instructions, thực hiện vòng lặp suy luận (Reasoning loop) và tạo lệnh gọi Tool.
Tool Sandbox Runner Phân rã tham số, chạy Tool ngoài (Midjourney, Web Scraping) trong môi trường an toàn (Sandbox) với Timeout nghiêm ngặt (30s).
[Self-Correction Loop] Vòng lặp tối ưu hóa quay lui nếu Evaluator phát hiện lỗi
Evaluation Check
Giai đoạn 4: Quality Check & human-in-the-loop (PAL Out)
Evaluator Critic Node Chấm điểm nhận diện thương hiệu (Brand Score). Nếu < 80%, kích hoạt tối ưu lại (Loop-back).
Human Approval Gate Điểm dừng chờ con người phê duyệt (trạng thái Pending) trước khi giải phóng Payload.
Execution Log Recorder Ghi lại thời gian phản hồi, chi phí token và trạng thái các node phục vụ kiểm toán.

2 PHẦN 2: 10 GIẢ THUYẾT VỀ CƠ CHẾ CHẠY NGẦM & BẰNG CHỨNG THỰC NGHIỆM

Danh sách 10 giả thuyết kiến trúc ngầm được suy luận khoa học từ các chỉ số đo lường như độ trễ (latency), thông báo lỗi (errors), cấu trúc JSON phản hồi, và hành vi trên giao diện (UI affordances).

Orchestration Loop (H1)

Bản chất xử lý tuần tự kết hợp bất đồng bộ (Sequential Dispatch)

Giả thuyết: Quá trình xử lý đầu vào diễn ra tuần tự qua các Node tiền xử lý trên gateway, nhưng việc thực thi tác vụ AI nặng (sinh ảnh, cào web) trên PAL được thực hiện bất đồng bộ (Asynchronous) để giải phóng tài nguyên mạng.

Bằng chứng (Evidence): Nhãn trạng thái (Status Labels) trên UI thay đổi liên tục theo trình tự thời gian: "Tối ưu brief..." (1-2s) → "Đang sinh ảnh..." (10-12s) → "Đang thẩm định..." (2s).
Orchestration Loop (H2)

Vòng lặp tự sửa đổi của Evaluator (Self-Correction Loop)

Giả thuyết: PAL Workflow hỗ trợ vòng lặp quay lui (Backtracking). Nếu Node Evaluator đánh giá chất lượng ảnh đầu ra không đạt chuẩn nhận diện, nó sẽ tự động kích hoạt chạy lại (Retry) Node Generator với prompt đã được tinh chỉnh.

Bằng chứng (Evidence): Trong một số phiên chat phức tạp, độ trễ sinh ảnh tăng đột biến lên 30s-35s (gấp đôi bình thường) và Execution Logs ghi nhận dòng trạng thái: "Evaluator rejected - retrying generator node with revised prompt".
Tool Interface (H3)

Đóng gói tham số Sandbox độc lập (Isolated Executions)

Giả thuyết: Các Tool (Midjourney API, Web Scraper) chạy trong môi trường Sandbox cô lập. Agent phân tách tham số từ prompt, đóng gói thành đúng schema của Tool, chạy độc lập và trả kết quả thô về để Agent parse lại.

Bằng chứng (Evidence): Khi cấu hình sai API Key của Tool trong PAL, lỗi xác thực 401 Unauthorized của nhà cung cấp bên thứ ba được ném thẳng lên cửa sổ log của Node đó chứ không làm sập luồng điều phối chính.
Tool Interface (H4)

Ngưỡng ngắt độ trễ nghiêm ngặt (Strict Timeout Limit)

Giả thuyết: Để bảo vệ hệ thống khỏi bị treo do API bên ngoài phản hồi chậm, bộ điều phối PAL áp dụng giới hạn thời gian chờ (Timeout) tối đa là 30 giây cho mọi thao tác gọi Tool.

Bằng chứng (Evidence): Khi gọi công cụ cào web trên một trang web bị chặn (lagging), Node Tool luôn trả về thông báo lỗi "Timeout Error: Node execution exceeded 30000ms limit" tại đúng mốc 30 giây.
Context & Memory (H5)

Bộ nhớ ngữ cảnh trượt phía Client (Client-Side Sliding Window)

Giả thuyết: PAL là Stateless Engine nên không lưu trạng thái hội thoại. PAM đảm nhận việc lưu lịch sử chat và tự động đính kèm `N` tin nhắn gần nhất (Sliding Window) vào Payload gửi sang PAL để Agent hiểu ngữ cảnh.

Bằng chứng (Evidence): Khi chat trên PAM, các câu lệnh ngắn như *"làm tối nó đi"* hoạt động tốt, nhưng nếu người dùng bấm **New Session** (tạo phiên mới), Agent lập tức quên toàn bộ hình ảnh và lịch sử của phiên trước.
Context & Memory (H6)

Tiêm tri thức động qua RAG (Semantic RAG Injection)

Giả thuyết: Khi gõ các thẻ `@-mention`, hệ thống thực hiện tìm kiếm ngữ nghĩa (Semantic search) trên Vector DB của Brand, sau đó tiêm (Inject) đoạn văn bản guidelines đó trực tiếp vào phần System Instruction của LLM payload.

Bằng chứng (Evidence): Gõ prompt bình thường phản hồi rất nhanh (1-2s), nhưng khi gõ thêm thẻ `@logo_vertical`, độ trễ gửi tin tăng thêm ~200ms (cho truy xuất DB) và cấu trúc payload ghi nhận trường "brand_context" được điền đầy đủ.
Guardrails (H7)

Kiểm duyệt an toàn ở lớp Gateway (Lightweight Input Filter)

Giả thuyết: Rào cản đầu vào không dùng LLM để tiết kiệm chi phí. PAM/PAL chạy bộ lọc từ khóa (Regex hoặc mô hình phân loại cực nhẹ) trực tiếp tại API Gateway để chặn nội dung nhạy cảm/độc hại từ sớm.

Bằng chứng (Evidence): Nếu nhập prompt chứa từ khóa cấm, lỗi "Chính sách nội dung vi phạm" trả về ngay lập tức (< 150ms), nhanh hơn rất nhiều so với độ trễ tối thiểu của một lượt gọi LLM thông thường (~1000ms).
Guardrails (H8)

Kiểm duyệt đầu ra bằng mô hình Vision (Output LLM Evaluator)

Giả thuyết: Để kiểm duyệt chất lượng hình ảnh, PAL chạy một mô hình đa phương thức (Vision LLM) riêng biệt ở cuối pipeline, phân tích metadata và pixel của ảnh để so sánh với Guideline đã nạp.

Bằng chứng (Evidence): Payload phản hồi thành công trả về cho PAM chứa một object có cấu trúc: "evaluation": { "passed": true, "brand_score": 92.5 } đi kèm lý do chấm điểm bằng ngôn ngữ tự nhiên.
Human-in-the-loop (H9)

Cơ chế ngắt dừng trạng thái (State-Persisted Pause & Gate)

Giả thuyết: Khi gặp Node Human Input, PAL lưu lại toàn bộ trạng thái chạy hiện tại vào database, đóng phiên chạy tạm thời và mở một Webhook Listener chờ tín hiệu HTTP POST phê duyệt từ Admin để kích hoạt chạy tiếp đồ thị.

Bằng chứng (Evidence): Workflow dừng lại ở trạng thái "Pending approval" vô thời hạn, không tiêu hao RAM/CPU chạy ngầm, và lập tức chuyển sang trạng thái "Resumed" ngay khi Admin bấm nút phê duyệt trên giao diện quản trị.
Human-in-the-loop (H10)

Sự can thiệp thủ công độc lập ở Client (Post-Execution Manual Override)

Giả thuyết: Các chỉnh sửa thủ công (như cắt ảnh, xoay ảnh, thêm bộ lọc màu có sẵn) của người dùng trên PAM không kích hoạt lại công cụ sinh hay gọi lại PAL, mà được xử lý thuần túy tại trình duyệt của client.

Bằng chứng (Evidence): Khi người dùng bấm cắt ảnh hoặc áp bộ lọc màu trên PAM, tác vụ hoàn thành ngay lập tức (< 50ms), không có hiệu ứng xoay loading và không phát sinh bản ghi chạy mới trong Execution Logs của PAL.

3 PHẦN 3: WRITE-UP & TECHNICAL INFERENCES (Phân tích chuyên sâu)

Phân tích Kiến trúc Harness & Các Quyết định Đánh đổi (Technical Trade-offs)

Thiết kế bộ khung thực thi (Harness) của PAM và PAL thể hiện một tư duy tối ưu hóa hạ tầng và trải nghiệm người dùng rất rõ rệt:

  • Đánh đổi giữa Stateless Engine & Stateful Client: Việc giữ PAL hoàn toàn không lưu trạng thái (Stateless) giúp hệ thống mở rộng (Scale-out) cực tốt. Khi hàng nghìn người dùng PAM chat đồng thời, PAL chỉ cần nhận JSON payload độc lập, xử lý DAG rồi trả kết quả về. Điểm đánh đổi là PAM phải chịu trách nhiệm đóng gói toàn bộ ngữ cảnh lịch sử chat gửi đi ở mỗi lượt. Điều này làm tăng kích thước payload truyền qua mạng nhưng bảo vệ tài nguyên RAM/Database của Backend PAL khỏi bị nghẽn.
  • Cơ chế Tự sửa lỗi (Self-Correction Loop) vs Chi phí API: Tích hợp Node Evaluator tự động kiểm duyệt hình ảnh và kích hoạt tối ưu hóa lại giúp cải thiện tỷ lệ thành công của tác vụ ngay từ lần chạy đầu tiên (First-time-right). Tuy nhiên, cái giá phải trả là chi phí gọi API (Midjourney, LLMs) sẽ tăng gấp đôi hoặc gấp ba khi vòng lặp quay lui được kích hoạt, đồng thời đẩy độ trễ phản hồi phía người dùng lên cao. Đây là sự đánh đổi giữa chất lượng sản phẩm đầu ra và chi phí vận hành/độ trễ.
  • Rào chắn an toàn Đa lớp (Multi-layer Guardrails): Hệ thống kết hợp bộ lọc Regex cực nhanh ở Gateway đầu vào và mô hình Vision LLM nặng ở đầu ra. Thiết kế này giải quyết bài toán hiệu năng: lọc bỏ nhanh các prompt vi phạm chính sách hiển nhiên mà không tốn chi phí gọi LLM, đồng thời vẫn đảm làm việc được chất lượng ngữ nghĩa và nhận diện thương hiệu phức tạp của hình ảnh đầu ra.
