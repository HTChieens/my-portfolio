Ràng buộc Mối quan hệ giữa các Thực thể (Entity Relationships)

1. Mối quan hệ nội bộ PAM (Creative Application)

Thực thể nguồn Thực thể đích Lượng số (Ratio) Logic Nghiệp vụ (Business Logic)
User / Member Brand N : M Một Thành viên thuộc nhiều Thương hiệu với các vai trò khác nhau (Marketer, Creative Manager).
User / Member Session 1 : N Một Thành viên khởi tạo nhiều phiên làm việc để thiết kế các chiến dịch quảng cáo.
Brand Playbook 1 : N Một Thương hiệu có thể thiết lập nhiều Kịch bản (Playbook) mẫu để hướng dẫn sinh ảnh định kỳ.
Brand Connector 1 : N Một Thương hiệu có thể cấu hình nhiều cổng kết nối (Connector) để kéo dữ liệu chiến dịch hoặc chia sẻ file.
Playbook Session 1 : N Một Kịch bản có thể được tải vào nhiều Phiên làm việc khác nhau của người dùng để áp dụng cùng bộ quy tắc.

2. Mối quan hệ nội bộ PAL (AI Workflow Engine)

Thực thể nguồn Thực thể đích Lượng số (Ratio) Logic Nghiệp vụ (Business Logic)
Workspace Workflow 1 : N Một Không gian làm việc chứa nhiều luồng xử lý AI tự động phục vụ các tác vụ khác nhau.
Workspace Agent 1 : N Một Không gian làm việc quản lý nhiều thực thể AI Agent chuyên biệt dùng chung trong các Workflow.
Workspace Tool 1 : N Một Không gian làm việc quản trị nhiều công cụ tích hợp (API, MCP, Composio) để chia sẻ chéo.
Workflow Execution Log 1 : N Mỗi lần chạy Workflow ghi lại lịch sử thực thi để theo dõi trạng thái, thời gian chạy và chi phí.
Workflow Channel / Deployment 1 : N Một Workflow có thể được triển khai công khai thành nhiều trang Web Form hoặc Chatbot widget nhúng.
Agent Node Tool N : M Một Node AI Agent có thể truy cập và sử dụng nhiều công cụ/API bổ trợ để thực thi hành động.
Trigger Workflow 1 : 1 Mỗi Webhook/API endpoint trigger ánh xạ và kích hoạt đúng một Workflow cụ thể bên PAL.
Workflow Node 1 : N Một luồng AI điều phối (SOP) được cấu thành bởi nhiều node logic và node hành động.
Node Node N : M Các node kết nối với nhau bằng các cung có hướng (Edges) tạo thành đồ thị xử lý (DAG).
Agent Node Agent 1 : 1 Mỗi Node LLM Agent thừa kế một cấu hình đặc tả Agent cụ thể (Prompt, System role, Model).
Agent Node Knowledge Source N : M Một Node AI Agent có thể truy cập nhiều tri thức bổ trợ (RAG) khác nhau trong cùng Vector DB.
Mối quan hệ liên kết liên hệ thống (Cross-System Linkages):
  • User / Member (PAM) ←[Đồng bộ SSO 1:1 / 1:N]→ Workspace (PAL): Thành viên định danh bên PAM (qua SSO V-ID) sẽ được phân quyền tương ứng để truy cập Workspace thiết kế Workflow bên PAL (nếu là Builder/Admin).
  • Brand (PAM) ←[Liên kết RAG 1:1 / 1:N]→ Knowledge Source (PAL): Mỗi thương hiệu (Brand) định nghĩa cấu hình guideline, hình ảnh chuẩn hóa bên PAM sẽ tương ứng với các tệp tin lưu trữ tri thức (RAG) bên PAL nhằm nhúng vào vector database khi gọi tác vụ.
  • Playbook (PAM) ←[Cấu hình tham số Workflow 1:1]→ Workflow (PAL): Kịch bản mẫu (Playbook) trong PAM ánh xạ tới một Workflow cụ thể bên PAL cùng với các tham số/Agent được định hình trước để sinh ảnh đồng bộ.
  • Connector (PAM) ←[Ủy quyền API 1:1]→ Tool (PAL): Cổng kết nối cấu hình trên PAM cung cấp Token/Credentials xác thực cho các Node Tool bên PAL thực thi gọi API (đọc dữ liệu chiến dịch Ads, đồng bộ Drive).
  • Design Brief (PAM) ←[Webhook Trigger 1:1]→ Trigger (PAL): Việc nhập Design Brief và gửi đi từ PAM kích hoạt một lần gọi Endpoint Webhook bên PAL để khởi động luồng Workflow.
  • Artifact (PAM) ←[Response Payload 1:1]→ Node (PAL Output/Evaluator): Node thẩm định và kết xuất bên PAL trả dữ liệu hình ảnh về để PAM render thành Artifact thô ban đầu hiển thị trên màn hình sáng tạo.
  • Session (PAM) ←[Theo dõi tiến độ 1:N]→ Execution Log (PAL): Mỗi phiên chat sáng tạo (Session) chứa lịch sử nhiều lần chạy workflow, tương ứng với các bản ghi Execution Logs bên PAL phục vụ giám sát tài nguyên tiêu thụ.

→ Click vào ảnh kết quả để mở Screen 3 (Inspector).

Màn hình xem ảnh gốc độ phân giải cao kèm bảng điểm đánh giá của Evaluator Node. Cung cấp các nút công cụ nhanh: Resize (1:1, 16:9), Upscale, Variant (tạo biến thể).

→ Ấn chỉnh sửa gửi lệnh chat mới và quay lại Screen 2.

Màn hình PAL (AI Orchestration Canvas)

1. Workflow Canvas (Thiết kế Luồng) Canvas chính

Giao diện kéo thả Canvas đồ thị trực quan (DAG). Người dùng cấu hình Trigger (Webhook), kéo thả các Node kết nối với nhau để định nghĩa SOP tạo ảnh tự động.

→ Double-click Node AI để mở Screen 2; Node RAG để mở Screen 3.
2. Agent Config Panel (Cấu hình AI) Chi tiết

Nơi cấu hình các Agent AI chuyên biệt: Nhập System Prompt, định nghĩa vai trò, chọn mô hình LLM nền tảng (GPT-4o, Claude) và điều chỉnh tham số sinh (Temperature, Top P).

→ Lưu cấu hình để đóng panel và cập nhật Node trên Screen 1.
3. Knowledge Source Manager (RAG) Cơ sở tri thức

Màn hình quản lý tài liệu tri thức thương hiệu bổ trợ. Cấu hình cơ chế Chunking size, Vector store Index. Nơi đây nhận dữ liệu đồng bộ tự động từ thư viện Brand bên PAM.

→ Liên kết Vector store ID tới Agent Node trong Screen 1.
4. Execution Monitor & API Settings Giám sát

Bảng điều khiển cho System Admin quản lý các Trigger Webhook, kiểm tra nhật ký chạy (Execution Logs) của từng Node, theo dõi chi phí API và đo lường thời gian phản hồi (Latency).

→ Tra cứu chi tiết lỗi của các Node để sửa đổi trên Screen 1.
Luồng tương tác liên màn hình (Handoff Screen Flow): Người dùng chọn Brand ở PAM Screen 1 → Vào PAM Screen 2 để chat nhập brief → Khi gửi brief, PAM kích hoạt chạy ngầm Workflow cấu hình bởi PAL Screen 1 → Workflow này lấy tri thức từ PAL Screen 3 cấp cho Agent cấu hình ở PAL Screen 2 → Trả kết quả ảnh về PAM Screen 3 để hiển thị & quản lý phiên bản. Giải thích Luồng Bàn giao (Hand-off Flow): PAM và PAL kết nối với nhau thông qua kiến trúc hướng sự kiện (Event-Driven):
  1. Người dùng tại PAM khởi tạo Brief, tích hợp tệp đính kèm và tag tri thức bằng thẻ @logo.
  2. PAM đóng gói yêu cầu thành file JSON gửi trực tiếp đến PAL Webhook Endpoint.
  3. PAL phân rã dữ liệu, truy xuất các tri thức bổ trợ (RAG), gọi các tác vụ và API mở rộng (Tool), sau đó thẩm định kết quả đầu ra (Evaluator).
  4. PAL phản hồi thành công và trả về URL tài nguyên đầu ra cho PAM để đóng gói thành Artifact và lưu trữ phiên bản mới.

3 PHẦN 3: WRITE-UP (Phân tích các Quyết định Thiết kế - Design Decisions)

1. Quyết định "Tách vỏ" (Decoupling Engine vs. Application)

Suy luận: Đội ngũ thiết kế giữ PAL là một nền tảng điều phối tổng quát (Agnostic AI Builder) và tách biệt hoàn toàn ứng dụng Creative Studio (PAM) làm giao diện chuyên biệt.

Lý do: Việc tách biệt này giảm thiểu Tải nhận thức (Cognitive Load). Một Marketer không cần (và không nên) nhìn thấy các Node logic phức tạp, Router, hay Webhook của PAL. Họ chỉ cần làm việc với khái niệm quen thuộc (Brand, Brief, Gallery). Thiết kế này giúp hệ thống dễ dàng mở rộng, hỗ trợ phát triển các App tiếp theo trên cùng lõi PAL mà không làm phình to mã nguồn.

2. Ẩn RAG và Context Injection qua UX @-mention

Suy luận: Tính năng @-mention trong PAM là phương án UX tuyệt vời để che giấu cơ chế kỹ thuật phức tạp của việc truy xuất tri thức bổ trợ (RAG) từ PAL.

Lý do: Thay vì bắt người dùng tải tệp tin thủ công lên cơ sở tri thức (Knowledge Base) và cấu hình biến, PAM mượn thói quen "tag tên" trực quan từ các mạng xã hội. Khi gõ @logo, hệ thống tự động liên kết tài liệu hướng dẫn thương hiệu tương ứng đưa vào Prompt ngầm, tạo nên một Frictionless UX mượt mà.

3. Kiến trúc State Management qua "Session" & "Version"

Suy luận: Workflow của PAL về bản chất hoạt động theo kiểu không lưu trạng thái (Stateless), trong khi PAM sinh ra các thực thể Session (Phiên làm việc) và Version (Phiên bản) để quản lý hội thoại.

Lý do: Ảnh tạo bằng Generative AI hiếm khi hoàn hảo ngay từ lần đầu. Thiết kế Versioning cho phép người dùng quay lui (Undo) dễ dàng, và cấu trúc Session duy trì bộ nhớ ngữ cảnh để hiểu lệnh chỉnh sửa tiếp theo (ví dụ: "làm tối ảnh này đi một chút"). PAM đảm nhận quản lý trạng thái, giải phóng tài nguyên tính toán thuần túy cho PAL.

4. Tính đóng - mở của hệ thống (Enterprise vs. SaaS)

Suy luận: PAL hoạt động như một SaaS mở rộng rãi hỗ trợ cổng thanh toán, trong khi PAM được cấu hình bảo mật cực cao với V-ID SSO và cam kết dữ liệu riêng tư hoàn toàn.

Lý do: Đây là quyết định phân luồng mô hình kinh doanh (Business Model). PAL tập trung thu hút khách hàng số đông (B2C/B2B nhỏ), còn PAM đóng vai trò giải pháp nội bộ Enterprise bảo vệ tối đa tài sản thương hiệu. Việc phân tách giúp đáp ứng các tiêu chuẩn tuân thủ (Compliance) khắt khe mà không cản trước khả năng tiếp cận rộng rãi của MindPal.

5. Trừu tượng hóa khả năng mở rộng (Playbook/Connector vs. Workflow/Tool)

Suy luận: PAM trừu tượng hóa các luồng AI phức tạp của PAL thành các khái niệm nghiệp vụ đơn giản là Playbook (Kịch bản chạy mẫu) và Connector (Cổng API liên kết ngoài).

Lý do: Người dùng PAM (Marketer) không cần biết cách lập trình Python hay xây dựng đồ thị DAG phức tạp trên PAL. Bằng cách đóng gói luồng AI thô thành Playbook và chuyển đổi thông tin xác thực thành Connector, PAM tạo ra một lớp bảo vệ tính toàn vẹn (Safety boundary) – cho phép người dùng mở rộng khả năng sáng tạo mà không thể làm hỏng logic vận hành của các AI Specialities nền tảng.

6. Vòng lặp Kiểm định Chất lượng Tự động (Evaluator Loop & Execution Log)

Suy luận: PAL thiết lập các Node Evaluator để chấm điểm và tự động tối ưu hóa (Optimizer Loop) kết quả sinh ảnh, kết hợp ghi chép Execution Logs chi tiết trước khi gửi phản hồi về PAM.

Lý do: Trong quy trình Generative AI, rủi ro lệch guideline hoặc lỗi hình ảnh là rất cao. Việc tích hợp cơ chế tự động đánh giá ở cấp độ Engine giúp sàng lọc ảnh lỗi ngay lập tức, tiết kiệm băng thông và dung lượng lưu trữ của PAM (chỉ lưu các ảnh chất lượng đạt chuẩn làm Artifact), đồng thời giúp System Admin dễ dàng kiểm soát chi phí token thông qua nhật ký thực thi.
